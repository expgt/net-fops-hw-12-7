### Задание 1

Основные преимущества различных методов масштабирования:


1. Активный master-сервер и пассивный репликационный slave-сервер

Архитектура с активным мастер-сервером и пассивным репликационным slave-сервером (master-slave) обеспечивает надежность и высокую скорость работы баз данных. 
Она позволяет распределять нагрузку на инфраструктуру и предотвращать потери информации.

Преимущества:
- Горизонтальное масштабирование чтения.
Позволяет перенаправить все поисковые и аналитические операции (SELECT) на slave-серверы. Это существенно снижает нагрузку на центральный master-сервер,    оптимизируя время отклика системы.
- Отказоустойчивость и непрерывность.
Наличие готовой реплики данных позволяет быстро переключиться на slave-сервер в случае аппаратного или программного сбоя мастера. Это предотвращает длительные простои сервиса (сокращает время Downtime).
- Безопасное резервное копирование.
Пассивный сервер дает возможность делать полные бэкапы данных hot-standby в фоновом режиме без блокировки основного узла и остановки обслуживания клиентских запросов.
- Простота управления и согласованность.
Наличие только одного узла, отвечающего за запись, исключает возникновение конфликтов изменения данных. Все транзакции (изменения, удаления) происходят предсказуемо и гарантируют согласованность информации.
- Изоляция аналитики.
Обеспечивает возможность запуска тяжелых, ресурсоемких отчетов на slave-узлах без ущерба для скорости обработки транзакций пользователей на главном сервере.


2. Master-сервер и несколько slave-серверов

Масштабирование по схеме «один master — несколько slave» (репликация) — это классический подход к распределению нагрузки в базах данных. 
Он обеспечивает высокую надежность и оптимален для проектов с преобладанием чтения.

Преимущества:
- Масштабирование операций чтения.
Slave-серверы принимают на себя все запросы на чтение данных, разгружая master-сервер. Это позволяет линейно увеличивать производительность при росте аудитории, просто добавляя новые slave-узлы.
- Повышенная отказоустойчивость.
Если master-сервер выходит из строя, данные остаются доступными для чтения с любого из slave-серверов. В большинстве систем один из slave-узлов можно оперативно повысить до статуса master.
- Резервное копирование без простоев.
Создание бэкапов можно выполнять на slave-сервере, не прерывая работу основного master-сервера и не создавая на нем дополнительную нагрузку.
- Геораспределенность.
Slave-серверы можно разместить в разных частях света (ближе к конечным пользователям), что значительно снижает задержки (latency) при выполнении запросов на чтение.
- Аналитика и фоновые задачи.
Тяжелые аналитические запросы или процессы генерации отчетов можно выполнять на изолированном slave-сервере, защищая master-сервер от падения производительности.


3. Активный сервер со специальным механизмом репликации — distributed replicated block device (DRBD)

Масштабирование с использованием активных серверов и механизма DRBD (Distributed Replicated Block Device) создает высоконадежную систему хранения с архитектурой shared-nothing. Она работает как сетевой RAID-1, синхронно или асинхронно зеркалируя блочные устройства между узлами на уровне ядра ОС Linux.

Преимущества:
- Нулевая потеря данных (RPO = 0).
Благодаря использованию протокола синхронной репликации (Protocol C), запись считается успешной только после того, как данные зафиксированы на дисках обоих серверов.
- Повышенная отказоустойчивость.
Выход из строя одного из физических серверов не останавливает работу сервиса — DRBD мгновенно переключает нагрузку на резервный узел без потери актуальности информации.
- Прямой доступ к диску.
В отличие от файловых сетевых хранилищ (например, NFS), DRBD работает на уровне блочного устройства, что обеспечивает высокую скорость ввода-вывода (IOPS) и отсутствие избыточных накладных расходов.
- Масштабируемость в режиме Active-Active.
Начиная с 9-й версии, DRBD поддерживает распределенные среды с одновременным доступом к данным с нескольких узлов. Это позволяет использовать совместно с DRBD распределенные файловые системы (такие как OCFS2 или GFS2).
- Поддержка динамической миграции виртуальных машин.
Активный режим работы позволяет реализовывать "живую" миграцию (live migration) виртуальных машин между узлами без остановки сервисов (например, в кластерах под управлением Proxmox).
- Горизонтальное масштабирование пула хранения.
Использование технологии в связке с менеджером томов LINSTOR позволяет развертывать масштабируемые программно-определяемые хранилища (SDS), объединяя десятки узлов в единую надежную сеть.
- Гибкость сетевой маршрутизации.
DRBD позволяет балансировать трафик репликации между несколькими сетевыми интерфейсами (TCP-сокетами), что устраняет узкие места и обеспечивает масштабируемость пропускной способности канала.


4. SAN-кластер

Масштабирование с помощью SAN-кластера (Storage Area Network) объединяет серверы и системы хранения данных в единую сеть. 
Основные преимущества включают гибкость (независимый апгрейд серверов и дисков), максимальную производительность благодаря выделенным скоростным сетям передачи данных (Fibre Channel, iSCSI) и бесперебойную работу критически важных сервисов.

Преимущества:
- Независимое масштабирование.
Вы можете наращивать вычислительные мощности (добавлять серверы) или объем хранилища (добавлять дисковые полки) отдельно друг от друга, не нарушая работу всей инфраструктуры.
- Высокая надежность (Fault Tolerance).
Если один из серверов выходит из строя, его задачи мгновенно подхватывают другие узлы кластера. При этом данные остаются в целости и доступности.
- Исключительная производительность.
Использование выделенных сетей SAN и протоколов блочного доступа (например, Fibre Channel) обеспечивает сверхнизкие задержки (latency) и высокую пропускную способность для тяжелых нагрузок (СУБД, виртуализация).
- Централизованное управление.
Единая точка управления ресурсами позволяет администраторам выделять дисковое пространство нужного объема «на лету» без физического вмешательства и переключения кабелей.
- Экономия ресурсов и пространства.
Консолидация данных позволяет отказаться от разрозненных локальных жестких дисков на каждом сервере, сокращая расходы на электроэнергию, охлаждение и обслуживание.

### Задание 2


План для выполнения горизонтального и вертикального шаринга базы данных: 

1. Вертикальный шардинг.

Таблицы делятся по столбцам. Этот метод эффективен, когда таблица становится слишком «широкой», а разные группы столбцов используются для разных задач.

Для выбранной схемы (Пользователи, Книги, Магазины) можно использовать следующий план вертикального шардинга: 

- Выделить логически несвязанные сущности и сформировать из них таблицы 
- Распределить данные таблицы на независимые серверы, тем самым снизив конкуренцию за ресурсы:

Server1:
- Таблица "customers" - выделены часто запрашиваемые поля first_name, last_name, email
- Таблица "books" - выделены нечасто запрашиваемые release_year, language, rating

Server2:
- Таблица "customers" - выделены нечасто запрашиваемые active и create_date
- Таблица "books" - выделены часто запрашиваемые поля title и description

Server3: 
- Таблица "stores" - для данной набольшой таблицы выделен отдельный сервер, что снижает нагрузку на другие серверы

При данной схеме серверы будут загружаны равномерно за счет сбалансированого распределения таблиц с часто и нечасто запрашиваемой информацией.

2. Горизонтальный шардинг.

Это подход, при котором таблица делится по строкам. Он идеально подходит для таблиц с огромным количеством записей.

Для выбранной схемы (Пользователи, Книги, Магазины) можно использовать следующий план горизонтальный шардинга:

Выбрать ключ шардинга:
- Таблица "customers" - customer_id
- Таблица "books" - book_id
- Таблица "stores" - store_id

Выделить диапазоны строк, которые будут распределены по серверам:
- Таблица "customers"
  Server1 - 1-4 rows
  Server2 - 5-8 rows
  Server3 - 8-10 rows
  
- Таблица "books"
  Server1 - 1-4 rows
  Server2 - 5-8 rows
  Server3 - 8-10 rows
  
- Таблица "stores"
  Server1 - 1-4 rows
  Server2 - 5-8 rows
  Server3 - 8-10 rows

При данной схеме серверы будут загружаны равномерно за счет сбалансированного распределенных диапазонов строк.
